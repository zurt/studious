# Improvement Recommendations — Code Review 2026-06-12

A full-codebase review of Studious (backend, frontend, CI, docs) against the
project's stated goals: a personal Japanese study tool, web MVP now / native
later, file-based storage, supply-chain-cautious, with quality benchmarks
guarding the transcription pipeline.

**Overall assessment.** The codebase is in good shape: clear module
boundaries (api / services / providers), consistent structured logging with
correlation IDs, an LLM audit trail with cost tracking, 86.8% backend and
90%+ frontend test coverage, and CI that runs tests plus security audits on
every PR. The findings below are mostly hardening and consistency issues,
not architectural problems. Items are split into **fixes applied on this
branch** and **items marked for review** — the latter need a judgement call
(cost, UX, or pipeline-benchmark implications) before acting.

---

## Fixes applied on this branch

### 1. Path traversal through resource IDs (security)

`doc_id`, `chapter_id`, `region_id`, and `job_id` from the URL were
interpolated directly into filesystem paths in `services/storage.py`. A
request like `DELETE /api/documents/..` resolved to
`data/documents/..` — the entire data directory — and
`shutil.rmtree`'d it. Reads had the same problem (arbitrary JSON files on
disk could be fetched by crafting an ID).

The app binds to localhost and has one user, so real-world exposure is low,
but the fix is cheap: storage path helpers now validate IDs against
`[A-Za-z0-9_-]{1,64}` and raise `InvalidIdError`, which the API maps to
404. All generated IDs are 12-char hex, so no existing data is affected.

### 2. File-descriptor leak on every upload

`upload_document` and `reupload_document` called `tempfile.mkstemp()` and
discarded the returned file descriptor, then opened the path a second time.
Every upload leaked one fd for the life of the process. Fixed by wrapping
the descriptor with `os.fdopen`.

### 3. Non-atomic `meta.json` writes and duplicated upload logic

Storage has `_atomic_write_text` (write-temp-then-rename) precisely so a
crash mid-write can't corrupt metadata, but `api/documents.py` bypassed it
with a bare `write_text` (plus an inline `import json`). The upload and
re-upload handlers also duplicated ~25 lines of suffix-classification and
temp-file logic. Added `storage.save_document_meta()` and shared helpers;
both handlers now use them.

### 4. Failed PDF render left a broken document behind

If `render_pdf_to_pages` raised (corrupt or password-protected PDF), the
document directory had already been created — the library then showed a
permanently broken zero-page document and the client got an opaque 500.
Uploads now clean up the partial directory and return 400 with the render
error. (Re-upload has a related but messier failure mode — see review
item R6.)

### 5. Untracked build artifacts

`frontend/coverage/` and `frontend/tsconfig.tsbuildinfo` are generated by
vitest/tsc and were showing up as untracked noise. Added to `.gitignore`.

### 6. GitHub Actions pinned by SHA (roadmap Phase 1.9 P2)

`actions/checkout@v4`, `actions/setup-node@v4`, and
`astral-sh/setup-uv@v3` were tag-pinned; tags are mutable and a compromised
action is exactly the supply-chain vector this project guards against.
Pinned all uses to full commit SHAs (with version comments so Dependabot
keeps updating them). This was already on the roadmap.

## Items marked for review — decide before acting

### R1. Deduplicate the job-failure boilerplate in `jobs.py` ⚠️

The five `_run_*_job` methods repeat the same ~15-line
fail-update-audit-emit sequence roughly ten times (~150 lines of
boilerplate). A `_fail_job(job_id, msg, *, audit_kwargs)` helper plus an
audit-context dataclass would roughly halve the file and make new job types
much cheaper to add.

**Why review first:** `jobs.py` is the transcription pipeline, and project
policy says pipeline changes warrant `make benchmark` — which spends real
API tokens. A pure refactor with the existing mock-provider tests is
probably safe, but it's your call whether the cleanup is worth the
benchmark run (or whether you'd accept test coverage alone as sufficient).

### R2. Sanitize VLM-generated markdown before `innerHTML` ⚠️

`chapter-view.ts`, `breakdown-pane.ts`, and `grammar-guide.ts` render model
output via `marked.parse(...)` straight into `innerHTML`. `marked` does
not sanitize — if a transcription ever contains `<script>` or an
`onerror=` attribute (a hostile textbook page is an unlikely but real
input), it executes. Hand-written UI strings are already escaped via
`escapeHtml`; only the model-output path is exposed.

**Why review first:** the fix is a new dependency (`dompurify`), which is
subject to the 7-day cooldown policy, and the threat model (your own VLM
output, local single-user app) may not justify it. If you take it:
`DOMPurify.sanitize(marked.parse(md))` at the three call sites, or a single
shared `renderMarkdown()` helper — which would also collapse the duplicated
parse logic.

### R3. Replace `alert()` error reporting with a toast/notice component ⚠️

~10 call sites use `alert()` for failures (upload, link, transcribe,
grammar guide). Blocking dialogs are jarring mid-study-session, and several
errors include raw URLs/status codes. A small non-blocking toast utility
(same pattern as `confirm.ts`) would improve daily use.

**Why review first:** pure UX preference; it adds frontend surface area to
a deliberately minimal vanilla-TS app, and you may simply not be bothered
by alerts in a personal tool.

### R4. Split `chapter-view.ts` (946 lines) ⚠️

It owns page navigation, region CRUD, linking mode, transcription detail,
job streaming, and toolbar wiring. Extracting the region-detail pane and
the link-mode state machine into modules (matching the existing
`modules/` pattern) would make it tractable. **Why review first:** large
mechanical churn in the file you edit most — worth doing deliberately, not
opportunistically, and ideally when no feature branch is in flight.

### R5. Job-file retention ⚠️

`data/jobs/*.json` grows forever (one file per transcription/breakdown
request, already ~hundreds). A startup sweep deleting terminal-state jobs
older than N days would cap it. **Why review first:** it deletes data, and
you may want the job history as a debugging record alongside the audit log.
An alternative is moving them under a dated subfolder so they stay greppable
but out of the hot path.

### R6. Re-upload failure handling ⚠️

`PUT /api/documents/{id}/file` deletes the old original and pages *before*
rendering the replacement. If the new file fails to render, the document is
left with stale metadata and no pages. Robust fix: render to a temp dir
first, swap only on success. **Why review first:** slightly more involved
change to a rarely-used endpoint; decide whether the endpoint matters enough.

### R7. Reuse provider instances ⚠️

`registry.get_vlm()` constructs a fresh `AnthropicVlm` (and a fresh
`anthropic.Anthropic` HTTP client) for every job. Caching the instance in
the registry would reuse connections. **Why review first:** touches
provider logic (benchmark policy), and the per-job cost is milliseconds —
mostly a cleanliness win.

### R8. Split `config.py` (540 lines) ⚠️

Settings, five long prompts, three tool schemas, and the pricing table
share one file. A `prompts.py` (+ `pricing.py`) split would help
navigation. **Why review first:** project policy flags "VLM prompts or
prompt configuration" changes for benchmarking; a pure move should be
benchmark-neutral, but every import site changes — bundle it with R1 if
you do both.

### R9. Repo-root working files ⚠️

`Chapter 1/2 Grammar Study.pdf`, `japanese_study_with_chatgpt.txt`,
`screenshots/`, and `AGENTS.md` sit untracked at the repo root. Decide per
file: commit (AGENTS.md?), relocate into `data/`, or gitignore. Not done
for you because they look like deliberate working materials.

### R10. Remaining Phase 1.9 P2 supply-chain items

Still open on the roadmap, in rough order of value: enable GitHub secret
scanning + push protection (repo settings, can't be done from code);
`deptry`/`depcheck` unused-dependency pruning; SBOM on release tags;
documented API-key rotation; sandboxed PDF rendering (PyMuPDF parses
untrusted PDFs in-process — rendering in a subprocess with a timeout would
contain a parser exploit).

### Minor notes (no action urgency)

- `preferences.save()` uses a bare `write_text`; could share the atomic
  writer. One-line risk, one-line fix.
- SSE: subscribing to an already-finished job yields only the `snapshot`
  event (the terminal `job-done` was emitted before subscription). The
  frontend appears to handle this via snapshot status; if you ever see a
  spinner stick after reload, this is where to look.
- `make clean`'s `backend/**/__pycache__` glob doesn't recurse under
  plain `sh`; harmless but ineffective.
- `submit_transcription` accepts any provider string and only fails at job
  runtime; validating against the registry at submit time would give a 400
  instead of a failed job.
- Settings could move to `pydantic-settings` `BaseSettings` instead of
  hand-rolled `os.environ` parsing — only worth it next time env handling
  grows.
